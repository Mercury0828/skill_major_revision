# Major Revision Response

Turn a decision letter, the reviewer comments, and the current manuscript into a complete revision package. The skill owns the full cycle: comment triage, new experiments where needed, the point-by-point response letter, the manuscript revision, and the consistency passes that keep all of it coherent.

## Deliverables

Every revision produces exactly three artifacts:

1. `response_letter.tex`: standalone article-class document with its own `fig/` folder and its own `response_refs.bib`. Built from `assets/response_letter_template.tex`.
2. `revised_manuscript.tex`: the clean revised paper. No markup, no colors.
3. `diff_manuscript.tex`: generated by latexdiff from the original and revised manuscripts. See "Change marking" below.

All three must compile to PDF before anything is declared done.

## Workflow

1. **Parse the decision letter.** Build a comment inventory: reviewer number, comment ID, the verbatim comment text. Do not paraphrase at this stage.
2. **Split compound comments.** A comment with several independently answerable asks becomes numbered sub-points (Comment 2 becomes 2.1, 2.2). See "Comment splitting" below.
3. **Triage every point.** Assign a stance (agree / partly agree / disagree with evidence) and a work type (writing-only / needs-experiment / infeasible, explain why). Match each point to an archetype in `references/comment-archetypes.md` and record the planned strategy in a triage table. Show the table to the user before drafting.
4. **Run the experiments.** For every needs-experiment point, write acceptance criteria first (expected trends, parameter ranges, the exact tables and figures the comment requires), then design and run the experiment until the results meet them. Validate your own results against the integrity checks before writing anything up. Full protocol: `references/experiments-for-reviewers.md`.
5. **Draft responses comment by comment.** One point at a time, in order, using the five-move anatomy below. Letter first, with full detail: tables, figures, derivations, no length limit. A thorough letter signals seriousness; aim for substance on every point, not padding.
6. **Revise the manuscript in step.** For each point, write the condensed manuscript change immediately after its letter response, insert it into `revised_manuscript.tex`, and check that it reads naturally in its surrounding context. The text quoted in the letter's revised block and the text in the manuscript must match verbatim.
7. **Cross-check coverage.** Every comment and sub-point has a response. Every response that claims a manuscript change points to a real location, and the quoted text matches the manuscript. No orphaned figures or labels in either document.
8. **Number-consistency sweep.** Every number appearing in the letter, the manuscript, the tables, and the figure captions traces back to the artifact that generated it. Re-derive paper numbers from the source data, not from intermediate logs. See `references/consistency-and-endgame.md`.
9. **Fresh-reader logic pass.** Read the revised manuscript front to back as a first-time reader, hunting for the seams that section-local edits create. See `references/consistency-and-endgame.md`.
10. **Self-red-team pass.** For each response, ask: would this reviewer be satisfied, or would they push back? Patch the weak responses before submission. See "Self-red-team" below.
11. **Build.** Run latexdiff, compile all three PDFs with the full multi-pass build, and inspect the diff PDF page by page.
12. **Endgame: page budget.** Only now reconcile the manuscript against the venue page limit. Any content cut from the manuscript at this stage forces a synchronized edit to the letter. See `references/consistency-and-endgame.md`.

Steps 1 through 3 end with a checkpoint: show the user the comment inventory and triage table and get confirmation before running experiments or drafting.

## Comment splitting

Split a comment when it contains independently answerable asks, especially when they need different evidence types (one sub-point needs an experiment, another needs a rewording). Splitting a complex comment shows the reviewer their concern was read carefully, and it keeps each response focused.

Do not split for the sake of splitting. A comment with one concern stays whole, even if it is long. A useful test: if two candidate sub-points would share the same response text, they are one point.

Number sub-points within each reviewer: Comment 1 with three asks becomes Comments 1.1, 1.2, 1.3. Keep these IDs stable for the rest of the revision and across rounds.

## Triage: verdict classes

Four verdicts, applied per point:

1. **Accept with experiment.** The objection changes what the evidence shows. Run the experiment (step 4). This is the default for any feasible experiment ask: if it can be done, do it.
2. **Accept as writing.** Existing results or theory already answer the concern, or the fix is wording, notation, structure, or a missing discussion. No new runs needed; the response shows where the answer already lives and what was clarified.
3. **Counter-strengthen.** The comment rests on a premise that can be refuted structurally, with evidence. Even here, open with a genuine concession (the premise was inviting because the paper was unclear), then present the evidence. The result should read as "your question made the paper stronger", never as "you are wrong".
4. **Decline with stated reason.** Rare. Only for asks that are genuinely infeasible (hardware access, research-scale extensions, structural constraints). Explain the principled reason in detail, offer the closest feasible alternative and actually do it, and frame the original ask as future work. Never decline on effort grounds.

The full archetype catalog, including "reviewer misread the paper" and "pure praise", is in `references/comment-archetypes.md`. Consult it for every point during triage.

## Response letter structure

Use `assets/response_letter_template.tex` as the starting file. It contains the complete preamble (colored block environments, float handling inside boxes, theorem environments) and the document skeleton. Do not rebuild the preamble from scratch.

Document order:

1. Title block: "Response Letter of [paper title]" plus the article number.
2. Opening paragraph: thank the editor and reviewers, summarize each reviewer's concerns in one sentence each, explain the letter's layout (comment, then response, then the revision made).
3. Response to the Associate Editor or Editor, including a "List of Main Modifications" enumeration that gives the editor a one-screen overview of the revision.
4. One section per reviewer, comments answered in order.
5. The letter's own bibliography in the reference block.

Per comment, three block types in fixed order:

- `commentblock` (white): the reviewer's comment, quoted verbatim, with its ID.
- `responseblock` (blue): the response, including any tables and figures that belong to it.
- `revisedblock` (light cyan), one or more: the exact revised manuscript text, each prefixed with a bold location header ("Revision in Experiments (Section VII-A):") and an italic one-line locator describing what was changed and where it sits.

## The five-move response anatomy

Every substantive response follows the same arc inside its `responseblock`:

1. **Thank and concede.** Open with thanks and a specific, genuine concession: "We thank the reviewer for pointing out this issue. We agree that the original manuscript did not explain X clearly enough." Find the real thing the reviewer is right about, even when the verdict is counter-strengthen. A generic concession ("we agree this is important") reads as evasive; name the actual gap.
2. **Diagnose the old version.** State precisely what the previous manuscript did and why that was insufficient or wrong. This shows the comment was understood at its root.
3. **Describe the fix in full detail.** For experiments: the complete setup (parameters, seeds, ranges) and the results, with tables and figures placed directly in the response block. For proofs: the actual corrected argument, not a summary of it. For wording: the reasoning behind the new formulation. Detail is the currency of a major revision response; when in doubt, include more.
4. **Interpret carefully.** State what the result does and does not show. If the fix involved weakening an earlier claim, say so explicitly: "We have removed the stronger wording about X and replaced it with this more precise statement."
5. **Point to the manuscript.** Close with "The specific correction has been made in Section N of the revised manuscript, as shown below." Then give the `revisedblock`(s).

Two modifiers on the arc:

- **Error admissions get an impact statement.** When conceding a real error (wrong proof step, wrong number, inconsistent notation), admit it plainly, fix it completely, and state whether the error touches the main results: "This correction does not affect the main results or the conclusions of the paper", or, honestly, what it does change and how the conclusions were updated. Reviewers and editors fear errors cascading into the conclusions; address that fear directly.
- **Pure-praise comments get a short acknowledgment only.** When a reviewer compliments something or explicitly requires no change, reply with one or two sentences of thanks. Forcing the five-move arc onto praise reads as sycophantic.

**Self-contained responses (reviewer comfort first).** Every response must stand on its own. The reviewer should never have to open another reviewer's response, an earlier comment, or the original manuscript to learn the substance of an answer or to see what the main change was. Restate the key numbers, evidence, and conclusions inline, and **duplicate the supporting figures and tables into each response that relies on them** — figure repetition across the letter is fine; making the reviewer hunt for the result is not. Cross-references are allowed only as an optional "see also" courtesy, never as the sole place the substance lives. The worst case is a **cross-reviewer pointer for substance** (an R2 point whose evidence is only in an R1 response): eliminate it by copying the figure and the numbers into the R2 point. The `revisedblock` already shows the manuscript change verbatim, so the reviewer need not consult the paper to know what was modified.

**Duplicate comments across reviewers:** the first occurrence and every later one each get a complete, self-contained treatment — the same figure and the same key numbers repeated — and may end with a courtesy cross-reference ("this is the same revision discussed for Reviewer 1, Comment 4"). Never answer with only "see our response above"; each reviewer reads only their own section.

## Writing principles

- **Low posture, sincere tone.** The reviewer spent unpaid hours on the paper. Every response acknowledges something they got right. Concede early and specifically; it costs nothing and buys credibility for the points where evidence pushes back.
- **Claims match evidence.** No overclaiming anywhere, in either document. If a reviewer flagged an overclaim, the new wording must be visibly more precise, and the response quotes both the removed and the replacement wording.
- **Rebut with evidence, never with assertion.** A disagreement is carried by an experiment, a derivation, or a structural argument, presented after a concession. If there is no evidence, there is no rebuttal; accept instead.
- **Scope limits are stated as plain facts**, not as defensive hedging: "This proposition does not claim additive composition for arbitrary joint measurements" is content. "It should be noted that we do not intend to claim..." is noise.
- **Style sweep on every batch of new prose.** Both the letter and every manuscript insertion go through `references/ai-style-checklist.md` before they are final. The checklist is per-batch: each revision round's new text gets its own pass. The checklist includes genre adjustments for response letters (explicit old-versus-new contrast is allowed there).
- Short declarative sentences. First person plural, active voice ("We added...", "We now state..."). Thank the reviewer once per response, not repeatedly.

## Letter versus manuscript: the asymmetry

The letter and the manuscript serve different readers and get different detail levels.

- **The letter gets everything.** Full tables, multiple figures per comment, complete derivations, sensitivity grids. There is no page limit on the letter; richness is evidence of effort. A response letter for a major revision should be long because its content is, with 20 or more pages per reviewer as the working scale for a heavy revision.
- **The manuscript gets the distilled version.** One condensed paragraph, at most one new table or figure per addition, chosen carefully: every manuscript figure costs page budget that must be reconciled in the endgame. New manuscript text must be self-contained and consistent with its surrounding sections, written as if it had always been there.
- **Captions differ by document.** Manuscript captions stay minimal: one sentence stating what is shown plus the conditions, takeaways in the body text. Letter captions can be richer and conclusion-first ("ASR drops from 0.96 to 0.54 as p reaches 0.5, confirming the noise parameter controls inference success"), because each letter figure is standalone evidence for a skeptical reader.
- The text quoted in a `revisedblock` and the text inserted into `revised_manuscript.tex` are identical, verbatim. This is checked in step 7.

## Experiments for reviewer comments

For any point triaged needs-experiment, follow `references/experiments-for-reviewers.md`. The short version:

1. Write acceptance criteria before running anything: the trends the result must show, the parameter ranges to cover, and the exact tables and figures the comment requires. The criteria are a self-check; if the first results miss them, debug or redesign and rerun until results and criteria agree, or until it is clear the expected trend was wrong, which is itself a finding to report honestly.
2. Prefer experiments that engage the reviewer's own framing. A reviewer-proposed baseline or setting is the best kind to implement: either it wins (their point improves the paper) or it loses to the full method (the delta is exactly the contribution).
3. Run the integrity checks on your own output before writing it up: values re-derived from source data, per-cell sanity checks on tables, plausible (irregular) digits, figures regenerated from tracked code and visually inspected.
4. **Improve before conceding.** When an experiment fails or a result is unfavorable, do not default to an honest caveat first. Diagnose the consequence (does it actually weaken the paper, or is it cosmetic?), then look for a *legitimate* improvement — a fairer or reviewer-proposed baseline, more data or seeds, a better metric, a representative regime instead of an adversarial corner case, sharper framing — and rerun. Fall back to an honest caveat only once improvement is exhausted or the trend is genuinely wrong. The red line is fixed: legitimate improvements only, never massage or fabricate data; and an integrity-sensitive reframe that moves a headline number is surfaced to the user before it is applied. (Example: an optimality-gap study that "blew up" on larger instances only because of an adversarial minimum-budget corner; measured at a representative budget it kept its headline, disclosed the corner as a stress case, and the by-product confirmed a proposition.) After the last comment, re-review every earlier honest caveat for an improvement now available.
5. Write up full detail into the letter, the condensed version into the manuscript.

## Change marking

**Default: latexdiff.** Keep `revised_manuscript.tex` clean and generate `diff_manuscript.tex`:

```
latexdiff original_manuscript.tex revised_manuscript.tex > diff_manuscript.tex
```

latexdiff often breaks on heavy math, tables, and TikZ. The fallback ladder, and the rule that a diff must never fail silently, are in `references/consistency-and-endgame.md`. Always compile the diff PDF and inspect it page by page; a mangled diff that "compiled" is still a broken deliverable.

**Exception: in-text marking.** Some editors require changes highlighted in the manuscript body instead of a separate diff file. In that case mark revised passages directly with `\hl{}` (package `soul`) or `\textcolor{blue}{}` and state the convention in the letter ("changes are marked in blue"). Default remains latexdiff; switch only when the venue asks.

## Multi-round revisions

Major revisions often run two or more rounds. Conventions that keep round 2 coherent:

- New letter file per round (`response_letter_r2.tex`), same template, same block system.
- Comment IDs stay stable across rounds. A round-2 comment about an earlier point references it explicitly: "As noted in our response to Comment 2.1 in the previous round...".
- Back-references are self-contained: summarize what was changed previously in one or two sentences before addressing the new concern. The reviewer does not have the old letter open.
- The round-2 diff is generated against the round-1 revised manuscript (the version the reviewers last saw), not against the original submission.
- Carry the triage table forward: a round-2 push-back on a round-1 response usually means the response was weak; treat it as a new point with the round-1 verdict as context, and do not simply restate the old answer louder.

## Self-red-team

Before submission, one pass over the finished letter, response by response, asking for each: if I were this reviewer, would I accept this, or push back? Flag responses that

- concede vaguely without naming the actual gap,
- answer a different question than the one asked,
- rebut without evidence,
- promise a change the manuscript does not actually contain, or
- leave the reviewer's strongest sub-point unaddressed.

Patch the flagged responses. This is a light pass, not a full multi-reviewer simulation; its purpose is catching the two or three weakest responses while there is still time to strengthen them.

## Maintenance

This skill improves by absorbing lessons from real revision rounds. After each completed revision:

- New comment patterns and the strategies that worked go into `references/comment-archetypes.md` as dated entries.
- Style rules the owner adds or revokes go into `references/ai-style-checklist.md` with a dated `[rule]` tag, following the existing format.
- Workflow changes (new steps, reordered passes, new failure modes) are edited into this file directly; keep the body under roughly 500 lines and push detail into `references/`.
- Record each change in the repo CHANGELOG with the date and the revision project it came from.
